Node status

LEADER The single node currently coordinating all writes. All PUT and DELETE commands are routed here. Shown in gold with a pulsing ring. The number inside (t:N) is the current term — a logical clock that increments each election.
FOLLOWER Passive replica. Receives log entries from the leader and applies them in order. Can serve GET requests from local state. If it stops hearing from the leader, its election timer fires and it becomes a Candidate.
CANDIDATE Transitional state during an election. The node has incremented its term and is requesting votes from peers. If it collects a majority it becomes the new Leader; otherwise it waits and retries.
PAUSED Simulates a network partition. The node is still running but ignores all Raft RPCs, making it invisible to the rest of the cluster.

The replicated log

Every committed write is an entry in the log. Each entry shows its index [N], the term it was created in (t:N), and the command. Entries only appear here once a majority of nodes have acknowledged them — that guarantee is what makes the store fault-tolerant.

Try this

Watch a re-election:
1.PUT a few keys so the log has some entries.
2.Click PARTITION on whichever node is currently the Leader.
3.Within ~400ms the other two nodes elect a new Leader.
4.PUT more keys — the cluster keeps accepting writes.
5.Click RECONNECT on the partitioned node — it catches up by replaying the log.
